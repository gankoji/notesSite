Synchronizing Disparate Data Sources

This is something that OneDrive, Google Drive, Dropbox, and many others are quite good at. Effectively, they're version control systems, applied to generic file storage rather than source code focus. I bring this up because I have a brand spanking new Samsung External Drive that I got to partner my (also brand spanking new) Apple M1 Macbook Air. That makes 4 external drives I have now T.T because I'm a genius like that.

How to manage these devices?

First cut is of course, to get rid of some of them. I don't need 4 external drives, at all. The reason I got the new drive was a) the sweet fingerprint reader for encryption key input, and b) the size. It's freaking tiny! It's wonderful to see how small these things have gotten, especially when we consider that it's a 2 TB SSD, so using it as a normal data drive isn't going to be painful at all. I think I'll keep the 4 TB 'desktop external' as a reasonable local backup and temporary option for os installs and such, and give the 4 TB usb powered one to Nigel once I wipe it. The 1 TB 'portable' can go to Scarlet to pair her laptop. Ok, so now I'm down to just one external drive that I need to carry with me regularly, and need to synchronize regularly.

Resilio Sync is a great option for this

It works out of the box without too much configuration, and as we've seen over the last few months, it works just fine on the RPi as well. So, is the answer to simply have sync running on the macbook and use it to kick back all of the external drive data to the NAS? Probably. On that note, I should probably think about setting up something like that solution for the kids' computers as well. Now that I think about it, if their code is open source, it's probably a great example of how to pull of microservices without C#!

Sadly, Sync is not open source. However, I was reminded of Apache NiFi, which is open source, and is definitely utilizing the microservices pattern in some pieces of the software for its scalability.
